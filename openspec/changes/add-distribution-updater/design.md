## Decisions

**D1 â€” Updater lives in `desktop-main/src/updater.ts`, invoked from `electron-entry.ts`.**
Matches the locked design spec. A small `initUpdater(store: ElectronStoreService)` function reads `enableAutoUpdate`; when falsy it logs one "updater disabled" line and returns without touching `autoUpdater`. Alternative â€” initializing inside a Nest provider â€” rejected: the updater is an Electron-shell concern like window creation, not part of the DI graph consumed by controllers, and keeping it out of `AppModule` means the plain-Node integration harness never loads it.

**D2 â€” Guard on `process.versions.electron` before importing `electron-updater`.**
Same pattern `ElectronStoreService` uses for `electron-store` (and per the memory note about the plain-Node integration server crashing on unguarded Electron imports). Use a dynamic `import('electron-updater')` executed only inside Electron so vitest/integration runs never evaluate the module.

**D3 â€” Flag as a top-level `enableAutoUpdate: boolean` in `AppStoreSchema`, absent â‡’ `false`.**
Extends the existing typed schema rather than a separate store or env var; the existing `get`/`set` surface and Map fallback come for free. Not a secret, so no SafeStorage involvement.

**D4 â€” `publish: github` in `electron-builder.yml`; publishing stays in CI, not electron-builder.**
`publish: github` makes electron-builder emit `app-update.yml` into packages so `electron-updater` has a feed. Release upload remains owned by `package.yml`'s `softprops/action-gh-release` job; guard against accidental auto-publish (electron-builder auto-publishes when `GH_TOKEN` is set on tag builds) by keeping `desktop:package` invoked with publish never (`--publish never` or `electron-builder --config â€¦ -p never` semantics) in the workflow/scripts. Alternative â€” generic S3/own-server feed â€” rejected: GitHub Releases is already the artifact home.

**D5 â€” Two docs pages, autogenerated sidebar.**
`docs/docs/install.md` and `docs/docs/code-signing-roadmap.md`; the Docusaurus sidebar is `{type: 'autogenerated'}` so no sidebar config change is needed. `setup.md` Option C gains a pointer to the install page; README gets a top-level Install link to the published site. Screenshots optional per #207 ("if reasonable") â€” text steps are the acceptance bar.

Master artwork â‰Ą1024Ă—1024 (operator supplied/approved; placeholder acceptable) lives in `build/`; derived `icon.icns` / `icon.ico` (16â€“256 px multi-size) / `icon.png` (â‰Ą512) are generated once (e.g. via `electron-icon-builder` or ImageMagick + `png2icns`) and committed rather than generated at build time â€” keeps CI simple and deterministic. electron-builder wiring: per-platform `win.icon` / `mac.icon` / `linux.icon` pointing into `build/` (electron-builder's default `buildResources` directory is `build/`, so the top-level convention also works, but explicit fields are self-documenting). ## Risks / Trade-offs

- [electron-builder auto-publishes when `publish` is configured and a `GH_TOKEN` is present on tagged CI builds, double-uploading against the `softprops` release job] â†’ pin `desktop:package` to `--publish never`; only the dedicated release job attaches artifacts.
- [Unsigned macOS builds make a flag-on updater fail Gatekeeper checks, confusing testers] â†’ keep default off; document in the code-signing roadmap that flag-on testing is only meaningful once signing lands (Windows/Linux can smoke-test earlier).
- [`electron-updater` import breaks the plain-Node integration harness or vitest] â†’ dynamic import gated on `process.versions.electron` (D2) plus unit tests covering the non-Electron path.
- [Zero-outbound-traffic acceptance is hard to prove positively] â†’ verify by code inspection + unit test that `checkForUpdates` is never invoked when the flag is false; optionally confirm with a packaged-app run and network capture, documented in the PR.
- [Placeholder icon ships and gets mistaken for final branding] â†’ #275 explicitly scopes artwork swap as a non-breaking follow-up; record placeholder status in the PR and icon README note.
- [`.ico`/`.icns` generation tooling differs across contributor machines] â†’ commit the derived assets; regeneration steps noted alongside them so the pipeline is reproducible but not build-critical.

## Migration Plan

No runtime migration. The store schema addition is additive (absent key reads as `false`). Rollback of any PR is a plain revert â€” no persisted-data or infra changes. Enabling auto-update for real users is a future change (post-signing) that flips the default or adds a UI toggle.
